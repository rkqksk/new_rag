# 동적 문서 로드 가이드 (Dynamic Loading Guide)

## 1. 개념: JIT (Just-In-Time) 문서 로딩

**핵심 원칙**: 모든 문서를 한 번에 로드하지 않고, 사용자 요청에 따라 **필요한 문서만 선택적으로 로드**합니다.

### 왜 필요한가?
- **토큰 효율성**: 200K 토큰 중 불필요한 문서로 낭비 방지
- **응답 속도**: 관련 없는 컨텍스트 제거로 빠른 처리
- **컨텍스트 품질**: 집중된 정보로 정확한 답변

---

## 2. 키워드 → 문서 매핑 테이블

| 사용자 키워드 | 필수 로드 문서 | 선택 로드 |
|--------------|---------------|----------|
| API, 엔드포인트, REST | `API.md` | `TESTING.md` |
| 성능, 최적화, 속도 | `PERFORMANCE.md` | `MONITORING.md` |
| 배포, Docker, CI/CD | `DEPLOYMENT.md` | `SECURITY.md` |
| 테스트, 단위테스트 | `TESTING.md` | `API.md` |
| 보안, 인증, 권한 | `SECURITY.md` | `API.md` |
| 모니터링, 로그 | `MONITORING.md` | `PERFORMANCE.md` |
| 데이터베이스, 스키마 | `DATABASE.md` | - |
| 프론트엔드, UI | `FRONTEND.md` | - |

## 3. 시나리오별 로드 예제

### 시나리오 A: "API 엔드포인트 추가하고 싶어요"
```
로드 순서:
1. core.md (항상 기본)
2. API.md (필수)
3. TESTING.md (API 변경 시 테스트 필요)
```
**예상 토큰**: ~15K

### 시나리오 B: "성능 최적화가 필요해요"
```
로드 순서:
1. core.md
2. PERFORMANCE.md (필수)
3. MONITORING.md (성능 측정 방법)
4. DATABASE.md (쿼리 최적화 시)
```
**예상 토큰**: ~20K

### 시나리오 C: "프로덕션 배포하고 싶어요"
```
로드 순서:
1. core.md
2. DEPLOYMENT.md (필수)
3. SECURITY.md (배포 전 보안 체크)
4. MONITORING.md (배포 후 모니터링)
```
**예상 토큰**: ~25K

## 4. 로드 우선순위 규칙

### Tier 1 (항상 로드)
- `core.md`: 프로젝트 전체 구조 파악

### Tier 2 (요청 기반 필수)
- 사용자 키워드와 직접 매칭되는 문서

### Tier 3 (연관 문서)
- 작업 완료에 도움되는 관련 문서

### Tier 4 (필요 시)
- 사용자가 명시적으로 요청한 경우만

---

## 5. 토큰 절약 효과

| 방식 | 평균 토큰 사용 | 절약률 |
|------|---------------|--------|
| 전체 로드 (기존) | ~80K | 0% |
| 동적 로드 (JIT) | ~18K | **77%** |

**실제 예시**:
- 단순 질문: core.md만 → 5K 토큰
- 복잡한 작업: core + 3개 문서 → 25K 토큰
- 전체 리팩토링: 필요한 모든 문서 → 60K 토큰

---

## 6. 로드 결정 규칙

### core.md만 로드하는 경우
- ✅ "프로젝트 구조 설명해줘"
- ✅ "어떤 기능들이 있어?"
- ✅ "기술 스택이 뭐야?"

### 추가 문서 로드하는 경우
- ✅ "코드 수정/추가" 요청
- ✅ "구체적인 구현 방법" 질문
- ✅ "문제 해결/디버깅" 요청

### 판단 기준
```
IF 사용자 요청 == "정보 조회":
    LOAD core.md only
ELSE IF 사용자 요청 == "작업 수행":
    LOAD core.md + 관련 문서들
```
